# LifeOnline 开发教训

## 2026-03-24

### 1. 本机先编译验证，再部署远程
- **问题**：跳过了本机 `npx tsc --noEmit` 编译检查，直接 commit + push + deploy 到 246
- **正确流程**：本机编译 → 本机 dev 调试 → 确认无误 → commit → push → deploy 到 246
- **规则**：任何前端改动必须先在本机通过 TypeScript 编译，再进行远程部署

### 2. 挂起的终端命令要及时清理
- **问题**：后台 git commit 命令用户无法 approve，导致阻塞
- **规则**：不要让需要用户交互的命令长时间挂在后台；如果命令挂起超过 2 分钟，主动终止并重试

### 3. 进度文档必须基于代码级检视，不能只看文档口径
- **问题**：PM 初次复盘基于上一轮文档口径标记了 P0 技术债务（SoulActionKind 重复定义），实际代码已经解决
- **正确流程**：读文档了解背景 → 读代码验证实际状态 → 再写进度文档
- **规则**：进度文档每次更新前，必须对关键锚点做代码级 grep/view 验证，不能仅凭记忆或上一轮文档口径

### 4. 标记完成前必须检查文件是否真实存在于磁盘
- **问题**：Sprint 3 中扩展了已有基础设施来实现推送功能，但 todo 上明确指定了 `[NEW] ExecutionNotificationReceiver.kt`，却未实际创建该文件，就标记了 [x]
- **正确流程**：完成编码 → 检查 todo 中每个标注 `[NEW]` 的文件是否在磁盘上存在 → 确认后再标记 [x]
- **规则**：标记任务完成前，必须用 `find_by_name` 或 `ls` 检查所有 todo 中所指定的 `[NEW]` 文件是否已实际落盘


### 5. API 规范与全量响应升级的策略
- **问题**：将原本的 `res.json(data)` 直接全量重构为 `res.json({ success: true, data })` 时，由于后端路由极多，容易破坏旧版前端解析。
- **规则**：任何涉及全局 `ApiResponse<T>` 规范的重构，优先修改并加固前端或客户端的 Fetch 取物器，使其具备向前兼容和向后兼容的防御性解析能力，再去替换服务端的返回包装结构。

### 6. SQL 拼接的静态扫描防呆
- **问题**：在 `db.prepare` 内部使用了字符串插值（如 `` `UPDATE table SET ${sets.join(',')} WHERE id=?` ``），即便参数 `sets` 是代码硬编码生成的（绝无注入可能），仍然会被安全静态扫描工具标记为漏洞。
- **规则**：为了可维护性和零警告的流水线（CI），对于 SQLite 的执行，放弃所有的字符串模板插值构建。即使逻辑冗长，也使用完全静态定义的 SQL 语句全字段更新执行，杜绝扫描告警。

## 2026-03-25

### 7. 服务端 API 响应格式变更后必须同步更新所有客户端
- **问题**：服务端 `ApiResponse<T>` 标准化后，所有响应被包装为 `{success, data: {...}}`，但 Android 端 `LifeOSService.kt` 仍按旧扁平格式解析顶层 key（如 `soulActions`、`actions`），导致 APP 三个列表页全部加载失败。
- **正确流程**：修改服务端响应格式 → 搜索所有客户端（Web、Android、Wear）中消费该 API 的代码 → 同步适配 → 编译验证
- **规则**：任何服务端 API 响应结构变更，必须 grep 全部客户端代码中引用该端点的位置，确认解析逻辑已同步适配。客户端应实现防御性解析（如 `unwrapApiResponse`），兼容新旧格式。

### 8. 外部执行器（OpenClaw等）的参数与超时设计
- **问题**：将只供 LifeOS 内部使用的归档维度信息（`outputDimension`）透传到了发往 OpenClaw 的 payload 中，容易引发远程系统的 422 参数校验拦截；另外基于 `fetch` 的 HTTP 请求默认 30s 超时对于外部物理动作（需长耗时）严重不足，导致任务被误判为失败。
- **规则**：
  1. 调用非自身域名的外部 API，严格**修剪 Payload**，仅保留对方所需的指令字段。
  2. 涉及“智能体执行”、“物理爬虫搜索”等不确定耗时的外部操作，如果在现有架构只能使用 HTTP 长轮询/同步等待，必须将超时 `DEFAULT_TIMEOUT_MS` 设置为一个极高值（如 10 分钟以上）以免中断。最优解应改为 Webhook 异步回调。

### 9. 警惕接口幻觉，善用大模型 Agent 的原生意图工具
- **问题**：在对接外部原生智能体平台 (如 OpenClaw) 时，前期在未查阅官方文档的情况下，凭直觉假想编造了一个 `/tasks/execute` 接口，直到用户提醒查阅官方源码才发现真正的标准 API 是 `POST /hooks/agent`。
- **神来之笔 (高阶用法)**：由于 OpenClaw 是一个大模型 Agent 系统，在发现它不支持通用 HTTP Webhook 返回通道时，无需舍近求远去给它写插件。**直接在发给它的提示词报文 (`message`) 末尾，嵌入强制性的系统级指令（System Directive）**，命令它在思考完成后使用自带的 Bash/Curl 工具主动将结果 POST 到我们期望的回调接口。
- **规则**：
  1. 对接任何外部开源/闭源项目前，**必须**查阅并核实真实接口及报文范式（Payload schema），绝不允许凭空幻觉捏造 Endpoint。
  2. 善用大模型自身的能力！如果对方是一个拥有工具链（Toolset）的智能体，最简单的回调设计不是改它的底层源码，而是直接在 Prompt 里以系统身份“命令”它执行 HTTP POST 将结果原样投递。

### 10. 警惕多智能体系统的“自激振荡反馈循环” (Self-Exciting Feedback Loop)
- **问题**：在实现了自动触发物理动作或生成提炼笔记后，系统（如 OpenClaw 回调、连续性提炼持久化）在 Vault 中生成了机器投递的 Markdown 文件。由于 Vault 的文件系统监听机制是全量无差别的，且之前的 Markdown 解析器 `parser.ts` 强制将不认识的类型统一回退为 `note`，这就导致了 `postIndexPersonaTrigger` 误认这些机器产物为用户的“新日记”，从而再次调用 LLM 解析、再次提取结论并生成灵魂审批卡片，引发了无限循环的“垃圾卡片轰炸”。
- **规则**：
  1. **状态(State)与产物(Artifact)隔离**：机器自动化任务的进行中状态、心跳日志、报错详情必须落盘在关系型数据库（如 SQLite 表），绝对禁止污染属于“人类知识库”范畴的 Vault 笔记文件夹。
  2. **构建认知触发免疫门控 (Immunity Gate)**：在事件驱动触发 AI 认知识别前，必须设置坚固的“免疫黑名单”。（A）建立目录隔离，拦截对如 `/assets/`，`/soul/continuity/` 和 `/system/` 目录下文件的分析；（B）在解析器 `parser.ts` 层级必须承认并放行机器产生的标识字段（如 `continuity_insight`, `agent_report` 等 `type`），以便触发器从元数据层面 `currentNote.type !== 'note'` 干净利落地完成拦截。

## 2026-03-26

### 11. 生产环境绝不可使用带有热重载机制的启动脚本
- **问题**：部署系统中的 systemd 服务配置（`lifeos-server.service`）错误地使用了 `pnpm dev`（绑定了 `tsx watch src/index.ts`）。当业务逻辑（如智体存储）或文件同步系统（如 Syncthing）触及后端文件系统时，`watch` 进程捕获到变动便立即重启服务端，导致生产环境出现无限退出的“重启死循环”，前端断连无法服务。
- **规则**：生产环境统一部署必须显式指向静态执行（如 `pnpm start` 引用的 `tsx src/index.ts` 或 `node dist/index.js`），严禁挂在任何带 `watch` / `nodemon` 参数的开发脚本上。

### 12. 提示词（Prompt）严格对齐上游系统产物的“字眼”
- **问题**：为了让服务端的 AI Orchestrator 强制提取手机端生成的“原始文字”，我们在 Prompt 中写死了“如果看到【原始文字】则提取”。但实际上，移动端 App 模板导出时的标准 Header 叫做 `## 原始转写` 或 `## 📝 文本捕获`。由于大模型匹配极为死板（或严谨），导致高潜力的任务评分全部垫底（0.2 分），引发调试深渊。
- **规则**：为多端协同系统编写 Prompt 时，**绝对不能凭借“生活口语”去假设文本标识**。必须抓取真实生成的物理文件（`.md`），将真正的分隔符、Header 或关键词原封不动地刻入 Prompt 中，确保精准狙击。

### 13. 本地缓存/文件系统的防抖与强制刷新（Memoization Bypass）
- **问题**：在远程调试 `FileWatcher -> Indexer -> AI DAG` 链路时，为了引发重试，习惯性地使用 Linux 的 `touch` 命令更新文件修改时间（`mtime`）。但 LifeOS 的文件系统索引（Indexer）中有极强的防测漏节流判断：`previousNote.content === currentNote.content`。由于 `touch` 未改变文件文本，系统直接阻断了 AI 调用，造成“为何代码已更新却不见 AI 执行日志”的幻觉。
- **规则**：在带有内容哈希（Content Hash）或文本级相等判断的成熟系统中构造触发测试时，**不要仅更新时间戳**。必须物理性地往文件末尾注入真实的、非不可见空白符的文字（如 `echo '\nTRIGGER' >> file.md`），以此穿透所有层级的缓存拦截器。

## 2026-03-28

### 14. 终端环境的网络代理隔离与 TLS 握手问题
- **问题**：在本地开发机执行代码同步（`git push` 到 GitHub）时，命令直接挂起并报出 `gnutls_handshake() failed: The TLS connection was non-properly terminated.`，此时浏览器却能正常秒开 GitHub，导致了对网络状态的误判。
- **原因**：Linux/Unix 终端环境（Terminal）与浏览器的网络链路是默认隔离的。科学上网客户端（如 Clash/V2ray）绝大多数情况下只接管了系统主代理（或被浏览器插件捕捉），但并不会自动代理终端命令行发出的请求。此外，命令行 `git` 编译时强依赖古板的 `gnutls` 库，在面对防火墙基于 SNI 的主动重置（RST）阻断时，几乎毫无韧性，直接报错阻断；而浏览器通常配有现代版的 SSL/QUIC 机制能智能重连接绕过。
- **规则**：**凡是遇到 git clone/push、npm install 报 TLS 握手断开、Socket 超时挂起等网络隔离特征时，第一时间确认代理端口，并在命令前精确注入环境变量**。绝对不要浪费时间盲目祈祷重试，应显式将命令行这列“绿皮火车”挂载到代理轨道上，例如：`https_proxy=http://192.168.31.1:7890 git push`。
