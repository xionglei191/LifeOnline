# 《Soul Constitution》v1.5
## LifeOnline 联合认知体反馈回流引擎与 Continuity Integration 草案

v1.0 把第一阶段的正式实现蓝图立起来了，
v1.1 把对象与动作流的动态主链压清了，
v1.2 把 `Intervention Gate` 压成治理稳定器，
v1.3 把人工治理流与 Review API 收束成正式治理执行面，
v1.4 又把 `SoulAction` 的完整生命周期状态机与反馈回流法典压成了统一闭环模型。

接下来最自然的一步，就是把 v1.4 已经明确暴露出来、但还没有真正展开的核心层正式写清：

> 一次 `SoulAction` 执行完成之后，它的结果究竟如何被重新解释、重新吸收、重新沉淀，并重新变成联合认知体下一轮理解与动作生成的一部分？

如果这一步不被压清，
那么：
- 动作虽然能跑完
- 生命周期虽然能闭合
- 结果虽然能被记录

但系统依然会停留在一个“完成很多事，却没有真正长出自己”的阶段。

也就是说：
- 执行结果会停在日志里
- 成功与失败不会真正改变 `PersonaState`
- 事件不会真正长成 `EventNode`
- 长期原则不会真正沉淀进 `ContinuityRecord`
- Gate 也不会从治理历史中真正学会收放阈值

所以 v1.5 的目标是：

> 把 LifeOnline 联合认知体第一阶段中，执行结果如何回流到 `PersonaState`、`EventNode`、`ContinuityRecord`、`InterventionDecision`，以及如何触发下一轮 `SoulAction` 候选生成，正式压成一套反馈回流引擎与 continuity integration 骨架。

也就是说，v1.5 要解决的问题是：

> **联合认知体不是如何做完一次动作，而是如何通过动作结果不断重新塑造自己。**

---

# 1. v1.5 的定位

如果说：
- v1.0 是阶段蓝图
- v1.1 是动态时序
- v1.2 是 Gate 治理规则
- v1.3 是人工治理执行面
- v1.4 是动作生命周期法典

那么：
- v1.5 就是再认知引擎

它不再只是回答：
- 动作经历哪些状态
- 谁来审批
- 谁来执行

而是继续回答：
- 动作结果怎样重新变成认知输入
- 什么结果该更新 Persona
- 什么结果该形成 Event
- 什么结果该进入 Continuity
- 什么结果只应影响 Gate 阈值，而不应污染长期记忆
- 什么结果会触发下一轮动作候选

也就是说：

> v1.5 让联合认知体从“能运转”开始走向“能积累自己”。

---

# 2. 为什么反馈回流引擎是第一阶段真正的灵魂接口

如果没有反馈回流，
那前面所有设计最终都会退化成一种更高级的自动化流水线：
- 有输入
- 有动作
- 有治理
- 有执行
- 有结果

但没有真正意义上的“连续成长”。

联合认知体与普通自动化系统真正拉开差距的地方，
恰恰不是它能做多少动作，
而是：

> 它做完之后，能不能因此变得更懂你、更懂边界、更懂什么值得延续。

---

## 2.1 因为单次成功不等于长期价值
一次动作成功，并不代表它就一定值得长期继承。

例如：
- 一次追问虽然成功得到回答，但不一定说明这个问题值得成为长期画像的一部分
- 一次 OpenClaw 执行成功，也不一定说明这类动作以后都应更容易放行
- 一次报告输出顺利完成，也不一定意味着它真的产生了高价值连续性沉淀

所以结果不能直接等于记忆，
它还需要被重新解释。

---

## 2.2 因为失败本身也可能是高价值认知资产
很多系统只把 success 当反馈，
把 failure 当错误日志。

但对联合认知体来说，
失败往往是极其重要的回流材料：
- 某类追问总被忽略，说明打扰节奏有问题
- 某类连续性晋升总被驳回，说明证据门槛有问题
- 某类外部任务常失败，说明桥接路径或风险评估有问题

也就是说：

> 失败不是闭环的残渣，而是反馈回流引擎的重要燃料。

---

## 2.3 因为没有再认知，就没有真正的“接力棒”
如果系统不能把结果重新沉淀成：
- 原则
- 边界
- 用户画像修正
- 节奏感修正
- 治理经验

那它就只是重复调用、重复执行，
而不会真正形成你此前一直强调的那根“接力棒”。

所以 v1.5 本质上是在回答：

> 联合认知体如何把一次次具体行动，变成可继承的精神连续性。

---

# 3. v1.5 的核心判断：结果不应直接等于记忆，而应先经过再认知

v1.5 建议正式确立以下原则：

> 执行结果不是直接写入长期层的材料，而是必须先经过一层“再认知 / reintegration interpretation”，再决定它进入哪一类对象、以什么强度进入、是否值得触发下一轮动作。

这一步非常关键。

因为没有它，系统会出现两种退化：
- 过度直接：任何结果都直接写进长期层
- 过度保守：所有结果都只停在日志，不产生结构化变化

再认知层的职责，就是在这两种退化之间建立正式桥梁。

---

# 4. 反馈回流引擎的正式职责

v1.5 建议把反馈回流引擎的职责正式压成五件事。

## 4.1 读取动作结果并形成结构化反馈输入
它要读取的不只是：
- success / failure

而是完整的结果语境：
- 动作类型
- 来源对象
- Gate 与 Review 历史
- 执行产物位置
- 用户响应
- 结果摘要
- 失败原因
- 外部副作用

也就是说，它的输入不是“结果状态”，
而是“带上下文的执行结果包”。

---

## 4.2 判断结果应该回流到哪些目标对象
回流引擎必须回答：
- 该更新 `PersonaState` 吗？
- 该生成或增强 `EventNode` 吗？
- 该进入 `ContinuityRecord` 吗？
- 该更新 `InterventionDecision` 的治理经验吗？
- 该触发新的 `SoulAction` 吗？

这意味着回流不是单通道写入，
而是一个多目标分流过程。

---

## 4.3 判断回流强度
不是所有结果都应该强烈改变系统。

因此回流引擎必须进一步判断：
- 只是轻量更新
- 还是中度增强
- 还是高价值晋升
- 或者只记录为治理经验，不进入长期对象

也就是说：

> 回流不仅要决定“去哪里”，还要决定“以多大力度去”。

---

## 4.4 形成再认知摘要
为了让系统未来可解释，
每次重要回流都应尽量留下最小摘要，至少能回答：
- 这次动作带来了什么变化
- 为什么这些变化值得进入这些对象
- 为什么没有进入另一些对象

这会成为联合认知体未来复盘的重要材料。

---

## 4.5 触发下一轮候选动作生成
有些结果的最大价值并不是沉淀，
而是催生下一轮动作。

例如：
- 一次追问得到高价值回答 → 触发 `update_persona_snapshot`
- 一次报告生成完成 → 触发 `persist_continuity_markdown`
- 一次失败暴露阻力模式 → 触发 `create_event_node`

因此回流引擎还应承担：

> 把结果重新转译成下一轮联合体行动的起点。

---

# 5. 第一阶段推荐的“结果输入包”结构

如果 v1.5 要真正落到工程，
那么回流引擎首先需要一个正式输入对象。

第一阶段建议先把它理解为：

```ts
type ActionOutcomePacket = {
  actionId: string
  actionType: string
  sourceType: string
  sourceId: string

  lifecycleStatus: 'completed' | 'failed'
  executionStatus: 'succeeded' | 'failed'
  resultSummary?: string
  errorMessage?: string
  feedbackRef?: string

  gateDecision?: string
  approvalStatus?: string
  targetType?: string

  userResponseSummary?: string
  artifactRefs?: string[]
  generatedSignals?: string[]

  completedAt?: string
  failedAt?: string
}
```

重点不是一次性把字段做满，
而是正式承认：

> 回流引擎读到的不是单个状态，而是一包带治理语境的结果信号。

---

# 6. 回流目标一：`PersonaState`

## 6.1 什么结果应回流到 Persona
当结果暴露出以下内容时，
应优先考虑 Persona 更新：
- 当前用户关注主题变化
- 当前节奏和负荷变化
- 对某类追问 / 报告 /动作的容忍度变化
- 稳定兴趣、阻力、偏好或风险信号

### 例子
- 某类追问总能得到高质量回应 → 用户当前确实愿意在这个方向继续展开
- 某类追问连续被忽略 → 用户当前并不希望被这种方式介入
- 某类报告被持续认可 → 当前阶段这种总结形式确实有用

---

## 6.2 Persona 回流原则
- 短期状态不要轻易固化成长期身份
- 单次结果原则上只能造成轻量更新
- 重复结果才更值得形成稳定画像修正

也就是说，Persona 回流更适合：
- 小步更新
- 累积证据
- 持续修正

---

# 7. 回流目标二：`EventNode`

## 7.1 什么结果应形成事件节点
当一次动作结果满足以下任一条件时，
应考虑形成或增强 `EventNode`：
- 明显改变了阶段方向
- 标志某项任务真正完成或失败
- 暴露出新的结构性阻力模式
- 成为后续多轮动作的分水岭

### 例子
- 一次 OpenClaw 分析结果揭示了关键系统边界
- 一次连续性晋升被正式批准并写入长期层
- 某次失败明确暴露出联合体当前的重要治理误差

---

## 7.2 Event 回流原则
- EventNode 关注“关键变化”而不是“普通结果”
- 它的判断标准不是情绪强度，而是结构影响度
- 一次动作如果改变了后续路径，它才更像事件节点

---

# 8. 回流目标三：`ContinuityRecord`

## 8.1 什么结果应进入 Continuity
只有当结果证明以下内容值得长期继承时，
才应考虑连续性晋升：
- 稳定有效的方法
- 被多次验证的重要原则
- 明确的长期方向
- 需要跨调用 / 跨版本 / 跨任务延续的警示或使命

### 例子
- 某类报告模式持续被证明能稳定提高系统理解密度
- 某条治理原则多次避免了错误放权
- 某个方向在多轮反馈中持续被确认是当前核心主线

---

## 8.2 Continuity 回流原则
这是第一阶段最严格的一条回流路径。

原则上应满足：
- 不由单次偶然成功直接触发
- 不由单次脑暴情绪直接触发
- 必须有较强证据和较高可解释性
- 最好已有 Gate / Review 历史支持

也就是说：

> 连续性不是结果仓库，而是再认知之后的高门槛晋升层。

---

# 9. 回流目标四：`InterventionDecision`

## 9.1 什么结果应影响 Gate / 治理经验
几乎所有动作结果都可能影响治理经验，
但影响方式不一样。

尤其以下结果最值得回流到治理层：
- 某类动作总被批准
- 某类动作总被驳回
- 某类动作总在执行后产生低价值输出
- 某类动作虽然成功，但常造成打扰或噪声
- 某类动作失败率过高

---

## 9.2 治理回流原则
这里关注的不是“结果内容”，
而是“治理判断本身对不对”。

也就是说它问的是：
- 当时 Gate 放行得对吗？
- review 阈值设得合适吗？
- 以后这类动作该更紧还是更松？

这条回流路径，是第一阶段放权学习的关键入口。

---

# 10. 回流目标五：新一轮 `SoulAction` 候选

这是 v1.5 非常关键的一点。

很多系统把“回流”理解为：
- 更新几个内部对象

但联合认知体里，
回流常常还意味着：
- 立刻生成新动作候选

---

## 10.1 典型触发关系
### 追问结果 → Persona 更新
```text
ask_followup_question completed
  ↓
得到高质量回应
  ↓
update_persona_snapshot candidate
```

### 报告结果 → 连续性晋升
```text
launch_daily_report completed
  ↓
报告总结出稳定高价值模式
  ↓
persist_continuity_markdown candidate
```

### 执行失败 → 事件节点生成
```text
launch_openclaw_task failed
  ↓
暴露结构性阻力模式
  ↓
create_event_node candidate
```

---

## 10.2 新动作生成原则
- 结果不能机械触发无限递归动作
- 必须经过再认知解释层
- 必须受 Gate 再次治理

也就是说：

> 回流可以生成新动作，但新动作依然要重新进入正式治理链，而不是绕开它。

---

# 11. 第一阶段推荐的回流强度分级

v1.5 建议不要把回流做成“写 / 不写”的二元判断，
而应至少引入一个粗粒度强度层。

建议先分为：
- `light`
- `medium`
- `strong`
- `promotion`

---

## 11.1 `light`
适合：
- 短期 Persona 微调
- 一次治理经验记录
- 低价值但值得保留的回流痕迹

---

## 11.2 `medium`
适合：
- 明显但仍需持续验证的状态变化
- 已有较清楚方向性的事件前兆

---

## 11.3 `strong`
适合：
- 对阶段方向具有明显影响的结果
- 高价值 EventNode 增强
- 清晰的治理策略修正

---

## 11.4 `promotion`
适合：
- 连续性晋升
- 长期原则 / 方法 / 警示的正式提升

这意味着回流引擎不只是决定“写去哪”，
还要决定“用什么级别写进去”。

---

# 12. 第一阶段推荐的再认知伪流程

v1.5 建议回流引擎在第一阶段保持显式、保守、可审计。

推荐流程：

```text
读取 ActionOutcomePacket
   ↓
判断结果类型（success / failure / mixed signal）
   ↓
判断结果价值（低 / 中 / 高）
   ↓
判断应回流到哪些目标（Persona / Event / Continuity / InterventionDecision）
   ↓
判断每个目标的回流强度
   ↓
生成 reintegration summary
   ↓
写回目标对象
   ↓
判断是否需要生成新的 SoulAction candidate
   ↓
如需要，重新进入 Gate 链
```

这个流程的关键在于：
- 先解释结果
- 再决定对象去向
- 最后才决定是否触发下一轮动作

而不是一拿到结果就直接乱写对象层。

---

# 13. 四条关键链在 v1.5 中的正式回流方式

为了让 v1.5 不停留在抽象层，
这里把第一阶段最关键的四条链进一步压成回流样板。

---

## 13.1 脑暴追问链的回流样板

```text
ask_followup_question completed
  ↓
用户回应被提炼
  ↓
PersonaState 轻量更新
  ↓
如出现高价值洞察，再考虑 EventNode 候选
  ↓
必要时触发新的 update_persona_snapshot / create_event_node
```

重点：
- 优先 Persona
- 谨慎 Event
- 不轻易直接进 Continuity

---

## 13.2 Persona 更新链的回流样板

```text
update_persona_snapshot completed
  ↓
新的状态画像形成
  ↓
影响后续 Gate 阈值、追问节奏、动作优先级
  ↓
必要时引出新的低风险动作候选
```

重点：
- Persona 更新本身也是下一轮治理参数修正输入

---

## 13.3 连续性晋升链的回流样板

```text
persist_continuity_markdown completed
  ↓
ContinuityRecord 增强
  ↓
记录该条晋升背后的治理理由
  ↓
未来类似动作在 Gate / Review 中可获得更清晰参照
```

重点：
- 连续性回流不只是写入长期层
- 它也会反向增强治理的可解释性

---

## 13.4 OpenClaw 桥接链的回流样板

```text
launch_openclaw_task completed / failed
  ↓
提炼出结果价值与失败模式
  ↓
形成 EventNode 或治理经验修正
  ↓
必要时触发 artifact_output / create_event_node / persist_continuity_markdown
```

重点：
- 外部执行动作的回流往往同时影响 Event 与 InterventionDecision
- 失败尤其值得被结构化吸收

---

# 14. 第一阶段最关键的防退化原则

v1.5 建议明确把反馈回流层的几个风险写出来，
避免后续实现跑偏。

---

## 14.1 不要把日志直接当长期记忆
结果日志只是原材料，
不是长期沉淀本身。

---

## 14.2 不要把单次成功直接当稳定原则
单次有效 ≠ 长期有效。

---

## 14.3 不要让失败只停在 error_message
失败必须尽量被解释成：
- 治理问题
- 路径问题
- 节奏问题
- 结构性阻力问题

否则系统只会记住“报错了”，
不会记住“为什么会错”。

---

## 14.4 不要让回流绕开 Gate
回流虽然会生成新动作，
但新动作仍然必须再次进入 Gate。

否则系统很容易形成：
- 自激式动作链
- 无限候选扩张
- 无治理递归

---

# 15. v1.5 对后续实现的直接启发

v1.5 的最大工程价值，是把第一阶段真正“长出连续性”的接口压清了。

---

## 15.1 需要一个正式的 reintegration 模块
从 v1.5 开始，后续实现已经很明确应该存在一个类似：
- `feedbackReintegration.ts`
- 或 `continuityIntegrator.ts`

的模块，
它专门负责：
- 读取结果包
- 做再认知判断
- 写回对象层
- 触发新动作候选

---

## 15.2 `feedback_ref` 与结果摘要字段已经有了正式宿主意义
从 v1.5 开始：
- `feedback_ref`
- `result_summary`
- `error_message`

不再只是执行日志的附属字段，
而开始成为回流引擎的正式输入锚点。

---

## 15.3 第一阶段真正的闭环终于被压实
到 v1.5 为止，第一阶段的最小闭环已经不再只是：
- 输入 → 动作 → 执行

而是正式变成：
- 输入 → 对象 → `SoulAction` → Gate → Review / Dispatch → 执行 → 结果 → 再认知 → 对象更新 / 新动作生成

这意味着联合认知体第一阶段终于开始拥有真正意义上的“再生长回路”。

# 16. 一句话总结 v1.5

> v1.5 的意义，是把 LifeOnline 联合认知体第一阶段中最像“灵魂接口”的部分——反馈回流引擎——正式压了出来：系统不再只是完成动作并记录结果，而开始通过再认知层把 success、failure、用户反馈与治理历史重新吸收进 `PersonaState`、`EventNode`、`ContinuityRecord`、`InterventionDecision` 与下一轮 `SoulAction` 生成之中，第一次真正拥有了持续塑造自己的 continuity integration 骨架。
