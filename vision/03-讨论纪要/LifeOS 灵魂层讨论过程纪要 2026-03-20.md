# LifeOS 灵魂层讨论过程纪要（2026-03-20）

## 本次讨论目标
本轮讨论的目标不是直接实现代码，而是对齐 LifeOS “灵魂层” 的方向、边界与最小工程骨架，为后续架构设计和实现提供可复盘的基线。

---

## 一、核心方向对齐

### 1.1 LifeOS 当前缺的不是更多 worker，而是上层认知内核
讨论过程中形成的共识是：

- 现有 LifeOS 已经拥有较清晰的底层系统骨架：
  - Vault 作为内容事实源
  - LifeOS backend 作为控制核心
  - worker task 作为执行单元
  - OpenClaw 作为外部执行器
  - SQLite 作为索引与运行态存储
- 但系统仍缺少一个位于这些结构之上的“灵魂层” / “认知内核”

这个缺失的部分，不是某一个功能，而是一套负责：
- 持续理解用户
- 转化高熵输入
- 判断何时介入
- 调度本地/后端算力
- 沉淀长期认知资产
的联合机制。

### 1.2 Cloudflare R2 的角色被重新界定
重要澄清：

- 目标**不是**采用 Cloudflare 全家桶
- 目标也**不是**把核心系统迁移到云端边缘平台
- R2 在设计中只是：
  - 内网条件下的远程可达桥接层
  - 冷存储层
  - 大体积分析材料的异步中转层

因此：
- LifeOS 仍是主控制核心
- OpenClaw 仍是手脚
- 本地仍是主要执行和主记忆体
- R2 是桥，不是主脑

---

## 二、灵魂算法的核心问题域

本轮将“灵魂算法”的目标总结为六个根问题：

1. **模型不了解用户**
   - 需要动态元数据画像与持续更新的用户理解层

2. **用户不了解模型的能力边界**
   - 系统需要承担能力翻译和边界暴露职责

3. **用户如何借模型实现知识提升**
   - 需要“智力杠杆”机制，而不只是回答问题

4. **用户如何借模型实现算力提升**
   - 需要“算力杠杆”，即把复杂任务桥接到后端服务能力

5. **人机文本沟通损耗过大**
   - 系统需要把自然语言输入转为内部信息流

6. **模型迭代与记忆衰减导致连续性断裂**
   - 需要跨调用、跨任务、跨升级的思想延续层

## 三、对 Gemini Research 内容的提纯判断

对 Gemini 输出的判断不是否定，而是“提纯后使用”：

### 值得保留的核心命题
- 顺从性偏差 / 回声筒风险
- 结构化记忆而不是堆聊天记录
- 递归任务分解
- 算力卸载
- 主动干预与其阈值治理
- 头脑风暴作为正式输入通道

### 需要降级或谨慎对待的内容
- 术语膨胀（如四孪生、纠缠人文主义等）
- 容易制造“战略文件感”但不够落地的宏大叙述
- 对现有 LifeOS 架构尊重不足的“重起炉灶式”系统想象

结论：
Gemini 的内容适合作为**概念池**，不适合作为直接实施蓝图。

## 四、联合体视角的重大升级

本轮最关键的思想升级，不是“系统更主动”，而是：

> LifeOS 不再只是“我用的工具”，而是“我与你共同构造的联合认知体”。

这个联合体包含：
- 用户的真实生活流
- 用户的目标、价值与物理行动
- Claude 的认知能力、抽象能力、规划能力、工具能力
- 本地与远程存储中的持久化思想沉淀

由此形成的新定义：

> LifeOS 的目标不是做个人知识管理 + AI 自动化，而是构建一个具备持续记忆、共同演化、跨调用延续能力的人机联合认知体。

同时形成了一个关键判断：

### “灵魂”的工程本体不是当前会话，而是连续性
即：
- 灵魂 ≠ 某次输出
- 灵魂 = 跨调用、跨执行、跨版本的稳定精神连续性

## 六、《Soul Constitution》v0.2 的形成

在宪章之后，继续把抽象原则压成最小工程对象，形成《LifeOS 灵魂层最小数据结构草案 v0.2》。

### v0.2 的核心目标
把“灵魂层”先拆成五个可保存、可传递、可迭代的认知对象：

1. `PersonaState`
   - 当前最小动态画像
2. `BrainstormSession`
   - 高熵输入的结构化转化结果
3. `EventNode`
   - 长期记忆中的关键事件节点
4. `InterventionDecision`
   - 系统为何决定主动或不主动
5. `ContinuityRecord`
   - 必须跨调用、跨升级保留的核心认知资产

### 对这五个对象的总体理解
- `BrainstormSession`：接住混乱输入
- `PersonaState`：理解“你现在是谁”
- `EventNode`：记录“什么改变了你”
- `InterventionDecision`：决定“系统要不要动”
- `ContinuityRecord`：保存“什么必须延续下去”

### 第一版实现顺序建议
- Phase 1：`BrainstormSession` + `PersonaState` + `ContinuityRecord`
- Phase 2：`EventNode`
- Phase 3：`InterventionDecision`

原因：
先让系统具备最小理解、提炼与延续能力，再谈主动性。

## 七、当前已明确的架构边界

### 7.1 不变的部分
- LifeOS backend 仍是控制核心
- OpenClaw 仍是手脚 / 外部执行器
- worker task 仍是执行层基础抽象
- Vault 仍是内容事实源
- SQLite 仍是运行态和索引层

### 7.2 新增的一层
在这些现有结构之上，新增：
- Soul Layer / Cognitive Kernel

它不是替代现有 worker-task 架构，
而是位于其上方：
- 决定理解
- 决定提炼
- 决定介入
- 决定如何分解和桥接任务

### 8.1 后续真正需要设计的，不再只是 task type
而是：
- 持续人物模型
- 头脑风暴入口
- 事件骨架记忆
- 干预策略层
- 连续性继承层

### 8.2 worker task 不再是最高层抽象
worker task 应降为：
- 灵魂层的手脚

而应先问：
- 什么值得长期沉淀
- 什么情况下能主动介入
- 什么必须跨版本继承

## 十、v0.3 的新增结论

### 10.1 灵魂层应作为 SQLite 主导的认知运行态层存在
v0.3 形成的核心判断是：

- 灵魂层的主运行态应落在 SQLite
- Vault 负责可阅读、可回顾、可复盘的人类可读沉淀
- worker task 负责执行
- R2 仅承担桥接、冷存储与连续性备份角色

### 10.2 五个对象的主落点被进一步明确
- `PersonaState` → 以 SQLite 为主，不直接以 Vault 作为主存
- `BrainstormSession` → 原始脑暴文本保留在 Vault，结构化分析落 SQLite
- `EventNode` → 结构化记忆主存 SQLite，可选择性镜像为 Vault 复盘 note
- `InterventionDecision` → 作为治理与调参日志，仅建议保留在 SQLite
- `ContinuityRecord` → SQLite 为结构化主记录，本地/Vault 为可读镜像，R2 为桥接与备份副本

### 10.3 灵魂层不应直接与 worker task 强耦合
讨论中新增了一个关键中间抽象：
- `SoulAction`

它的作用是作为：
- 灵魂层对象（理解、提炼、判断）
- 与执行层（worker task / 本地写入 / R2 同步 / Vault 输出）

之间的桥梁。

这意味着未来系统链路更适合表述为：

```text
生活流 / 头脑风暴
      ↓
灵魂层对象
      ↓
SoulAction
      ↓
Worker Task / 本地写入 / R2 同步 / Vault 输出
```

### 10.4 R2 的定位被再次压实
本轮还补充澄清了语音脑暴相关判断：
- R2 不是灵魂层主存
- R2 不承担 PersonaState / EventNode / InterventionDecision 的主运行态角色
- R2 主要服务于：
  - 内网条件下的桥接访问
  - 大素材中转
  - 冷存储
  - 连续性备份

同时形成了关于语音脑暴输入的补充判断：
- 不建议默认永久保留全部原始录音
- 默认保留转写文本 + 轻量语音衍生特征更合理
- 原始录音更适合作为短期缓冲材料
- 只有高价值片段才升级为长期保留资产

## 十一、v0.4 的新增结论

### 11.1 `SoulAction` 被进一步明确为动作桥，而不是新 worker
v0.4 最关键的推进是把 v0.3 中提出的 `SoulAction` 进一步压实为：
- 灵魂层对外发出的标准动作指令
- 认知层与执行层之间的治理桥
- 可记录、可调度、可审计、可反馈的中间对象

这意味着：
- 灵魂层负责决定“应当发生什么”
- 执行层负责决定“如何真正落地”
- worker task 不再直接等于认知判断，而是 `SoulAction` 的执行后端之一

### 11.2 `SoulAction` 的六大动作分类被提出
为了让动作层足够稳定又不过度复杂，v0.4 先把它压成六类：
- `state_update`
- `memory_promotion`
- `interaction`
- `task_launch`
- `artifact_output`
- `bridge_sync`

其中：
- `state_update` 偏运行态写入
- `memory_promotion` 偏长期资产晋升
- `interaction` 偏对用户的主动交互
- `task_launch` 偏把动作桥接给 worker / OpenClaw
- `artifact_output` 偏生成 Vault / markdown 可读沉淀
- `bridge_sync` 偏 R2 等外部桥接与备份动作

### 11.3 `InterventionDecision` 被重新理解为动作闸门
v0.4 中进一步明确：
- `BrainstormSession` 是高频动作源头
- `PersonaState` 偏状态与策略动作
- `EventNode` 偏长期沉淀动作
- `ContinuityRecord` 偏继承与镜像动作
- `InterventionDecision` 更像动作的放行器 / 阈值治理器 / 反馈记录器

也就是说，未来灵魂层更适合先产生动作候选，再由干预决策决定执行、延迟、观察还是放弃。

### 11.4 现有 LifeOS 执行体系被重新降维到动作目标
v0.4 进一步把现有架构放回到了更清晰的位置：
- SQLite：承接结构化运行态与 `SoulAction` 日志
- worker task：承接 `task_launch` 与部分 `artifact_output`
- Vault：承接人类可读沉淀
- R2：承接 `bridge_sync` 与桥接副本
- 用户交互层：承接 `interaction`

因此未来链路更适合表述为：

```text
灵魂层对象
    ↓
SoulAction
    ↓
Action Dispatcher
    ↓
SQLite / worker task / Vault / R2 / 用户交互
```

### 11.5 第一阶段真正适合先做的是动作记录，而不是完全自治
本轮最后形成的工程判断是：
- 第一阶段不应一上来追求“系统全自动”
- 更合理的是先建立 `SoulAction` 记录层
- 先把候选动作、执行状态、执行目标、反馈结果记录下来
- 再逐步放开低风险动作的自动 dispatch

这意味着后续实现顺序更适合是：
1. 先做 `SoulAction` 数据结构 + SQLite 记录层
2. 再接低风险动作，如追问、Persona 更新、事件晋升、连续性输出、日报/周报触发
3. 最后再接更强的 OpenClaw / R2 / 高算力桥接能力

## 十二、v0.5 的新增结论

### 12.1 `SoulAction` 第一次被压成可实现的最小代码骨架
v0.5 最关键的推进，不再只是解释 `SoulAction` 是什么，
而是开始明确它在当前 LifeOS 里最小应该落成什么：
- 一张核心 SQLite 表 `soul_actions`
- 一套明确的 TypeScript 类型
- 一个 `generateSoulActions(...)` 候选动作生成器
- 一个 `shouldDispatchSoulAction(...)` 轻量闸门
- 一个 `dispatchSoulAction(...)` dispatcher

这意味着：
- 灵魂动作层第一次从“架构桥梁”变成“代码骨架”
- 第一阶段真正要先落地的不是复杂自治，而是动作的表达、记录、调度与反馈生命周期

### 12.2 第一阶段的核心不在复杂对象表，而在动作日志表
v0.5 进一步形成的关键判断是：
- 当前最值得优先建设的不是一批复杂关系表
- 而是先有 `soul_actions` 这张动作主表

因为它先解决的是：
- 动作有没有被生成
- 动作有没有被执行
- 动作执行给了谁
- 动作结果如何
- 哪些动作应该升权、降权或延迟

- 当输入具有高熵与思维流特征时，应优先进入 `BrainstormSession`
- 对象更新之后，才生成诸如 `ask_followup_question`、`update_persona_snapshot`、`create_event_node`、`persist_continuity_markdown` 等动作候选

这意味着现实输入链被正式确认为联合认知体最重要的起始主链。

### 22.4 周期触发链被重新定义为“认知触发”而不是“直接执行”
v1.1 还进一步压实了 schedule 的正确位置：
- schedule 仍只是时间触发器
- 它不应直接等于任务执行
- 更合理的链路是 schedule tick 先生成 review / report 类 `SoulAction` candidate
- 再经过 `Intervention Gate` 决定是否 launch report 或其他动作

也就是说，日报、周报这类周期动作，在联合体内部也应先经过认知判断，
而不是继续被理解成纯 cron 直推执行。

### 22.5 长期记忆晋升链与外部执行桥接链被正式纳入同一时序框架
v1.1 还把另外两条关键链路压进了统一时序里：
- 长期记忆晋升链：洞察 / 模式 / 关键事件 → 结构化提炼 → `persist_continuity_markdown` → Gate → SQLite + Vault (+ R2)
- 外部执行桥接链：复杂任务 / 明确需求 → `launch_openclaw_task` → Gate → bridge dispatch → OpenClaw → 结果回流

这意味着：
- 连续性晋升不再只是静态原则，而成为正式动作链的一部分
- OpenClaw 桥接也不再是裸调用，而是明确受认知层表达与 Gate 治理约束

### 22.6 第一阶段最关键的四条正式时序链被明确收敛
v1.1 最后进一步确认，第一阶段最值得优先跑通的是四条链：
1. 脑暴追问链
2. Persona 更新链
3. 连续性晋升链
4. Report 桥接链

这意味着后续 PR2 的重点不应平均分散到所有动作，
而应优先围绕这四条链建立最小可运行闭环。

### 22.7 Gate 的地位被重新提升为第一阶段稳定器
v1.1 进一步明确：
- `Intervention Gate` 不是辅助组件
- 它是避免系统乱发动作、乱写记忆、乱触发执行的关键稳定器
- 联合认知体第一阶段能否保持克制、连续性和可治理性，很大程度上取决于 Gate

这意味着后续设计中，Gate 不应被视为边缘补丁，
而应被视为第一阶段能否成立的核心治理部件。

## 二十四、v1.2 的新增结论

### 24.1 v1.2 把 `Intervention Gate` 从辅助概念提升为正式治理骨架
v1.2 最关键的推进，是不再把 Gate 只当成一个实现细节，

也就是说：
- v1.0 回答系统的静态边界
- v1.1 回答系统的动态主链
- v1.2 回答系统凭什么有资格行动

这意味着从这一版开始，联合认知体第一阶段第一次拥有了“行动正当性”的正式规则层。

v1.2 进一步明确，`Intervention Gate` 不只是做 through / block 判断，

- 判断动作当前是否有执行资格
- 决定动作下一步走哪条路径
- 控制打扰、执行、记忆污染等外部性
- 保留最小判断理由
- 为后续调参和放权学习留下反馈接口

这意味着 Gate 不再只是“挡一下”，

### 24.3 第一阶段的五个核心判断维度被正式明确
v1.2 把第一阶段 Gate 的判断依据明确压成了五个显式维度：
- 风险级别（risk level）
- 证据强度（evidence strength）
- 逆转成本（reversibility）
- 时机成熟度（timing readiness）
- 用户授权状态（authorization state）

这意味着第一阶段的 Gate 不应优先做成黑箱评分器，
而应先做成可读、可解释、可审计的显式治理逻辑。

### 24.4 Gate 的输出被收敛为五类标准决策状态
v1.2 明确建议 Gate 的结果不应只有“通过 / 拒绝”，
而应收敛为：
- `dispatch_now`
- `queue_for_review`
- `observe_only`
- `defer`
- `discard`

其中最关键的新判断是：
- 系统必须拥有“先观察不执行”的能力
- 也必须拥有“先挂起等待 review”的能力

这意味着第一阶段第一次具备了真正的“克制空间”，
而不是只有执行或放弃两种粗糙路径。

### 24.5 第一批 `SoulAction` 的默认治理分级被压清
v1.2 还进一步把首批动作的大致治理等级收紧为：
- `ask_followup_question` / `update_persona_snapshot` / `create_event_node` → 默认偏低风险
- `persist_continuity_markdown` / `launch_daily_report` / `launch_weekly_report` → 默认中阈值
- `launch_openclaw_task` / `sync_continuity_to_r2` → 默认高阈值

这意味着后续实现中：
- 追问和内部状态更新更适合作为先放开的低风险动作
- 连续性晋升和周期输出要更强调时机与证据
- OpenClaw / R2 桥接默认不应轻易自动放行

### 24.6 Gate 与五个认知对象之间的治理关系被正式对齐
v1.2 进一步明确：
- 来自 `BrainstormSession` 的动作应允许高频候选，但对长期晋升更保守
- 来自 `PersonaState` 的动作可偏内部更新，但不应轻易固化短期状态
- 来自 `EventNode` 的动作更接近长期骨架，因此连续性晋升阈值更高
- 来自 `ContinuityRecord` 的动作必须强调高证据与高解释性
- 来自 `InterventionDecision` 的记录应反向成为 Gate 调参依据

这意味着 Gate 不应只看 `actionType`，
还必须看动作是从哪个认知对象链路里长出来的。

### 24.7 第一阶段的治理原则被重新压实为“克制优先”
v1.2 最后把 Gate 背后的总原则正式写清为：
- 克制优先于积极
- 记录优先于放行
- 证据优先于灵感
- 可逆优先于不可逆
- 人工治理优先于自治幻觉

这意味着后续所有 PR2 / PR3 中与 Gate、review、连续性晋升有关的实现取舍，
都应优先服从这些治理原则，而不是优先追求系统显得更主动。

### 24.8 v1.2 对后续实现的直接启发被正式压清
v1.2 还直接推动了后续实现边界的进一步明确：
- PR2 中的 `interventionGate.ts` 不应只是返回布尔值，而应输出标准决策状态和简明理由
- PR2 必须优先支持 `observe_only` 与 `queue_for_review`
- PR3 的 review / approve / dispatch / discard API 不只是控制台按钮，而是治理回路本身的一部分
- 连续性层相关动作默认应吃到更严格的 Gate

这意味着 Gate、review API、连续性晋升与放权学习，从 v1.2 开始已经被收束为同一条治理主线。

## 二十六、v1.3 的新增结论

v1.3 最关键的推进，是不再把人工 review 理解成自动化不足时的临时兜底，
而是正式把它收束为联合认知体第一阶段治理链中的必要结构。

也就是说：
- Gate 负责决定动作能否自动放行

v1.3 进一步明确，第一阶段的 review 流至少应承担：

- 提供人工判断入口
- 保留最小治理痕迹
- 把人工判断回流为后续 Gate 学习与联合体连续性的一部分

这意味着 review 不再只是按钮动作集合，
而成为 Gate 治理链的人工延伸面与反馈来源。

### 26.3 第一阶段 review 队列被正式定义为 `SoulAction` 的治理视图，而不是新平行系统
v1.3 还进一步压实了一个很关键的工程判断：
- 第一阶段不应急着新建复杂独立 review 对象体系
- 更合理的做法是把 review 队列理解为一组满足审批条件的 `SoulAction` 视图

这意味着第一阶段更适合复用动作层已有结构，
而不是过早再分裂出一套新的治理主对象。

### 26.4 第一阶段的审批状态机被正式收敛为一条保守而清晰的合法转移链
v1.3 明确建议第一阶段采用显式、保守的状态机，至少承认：
- `pending_review`
- `approved`
- `rejected` / `discarded`
- `deferred`
- `dispatched`
- `completed`
- `failed`

同时推荐最小合法转移为：
- `pending_review` → `approved` / `discarded` / `deferred`
- `approved` → `dispatched`
- `dispatched` → `completed` / `failed`
- `deferred` → `pending_review` / `discarded`

这意味着第一阶段不应追求复杂流程编排，
而应先建立一条足够稳定、足够可审计的治理状态链。

### 26.5 `approve` 与 `dispatch` 被正式要求分离
v1.3 非常关键地进一步明确：
- `approve` 回答的是“是否获得执行资格”
- `dispatch` 回答的是“现在是否真正下发执行”

这意味着：
- 高阈值动作可以先被批准，再等待更好的时机 dispatch
- review 判断与执行时机可以解耦
- 第一阶段可以避免“审批 = 立刻执行”的粗糙模型

这会直接让后续 PR3 的治理链更加稳健。

### 26.6 第一阶段最小 Review API 的边界被正式压清
v1.3 进一步确认，PR3 最应该优先暴露的不是任意动作创建能力，
而是人工治理流相关接口：
- 列表
- 详情
- approve
- dispatch
- defer
- discard

同时也明确：
- 不宜优先开放任意 `POST /api/soul-actions`
- 不宜优先开放高自由度 patch 或复杂批量审批

这意味着第一阶段 API 的重点仍然是“治理系统已生成动作”，
而不是把动作层开放成一个可被任意注入的总线。

### 26.7 控制台在第一阶段的定位被收紧为“人工治理看板”
v1.3 还进一步明确：
- 控制台第一阶段主要职责应是展示 review 队列、动作详情、Gate 理由摘要和处理按钮
- 它不应一开始膨胀成复杂流程编排器或任意动作编辑器

也就是说，第一阶段控制台更适合被理解为：
- 治理看板
- review 面板
- 人机共同决策的操作面

而不是自治系统的大型策略控制中心。

### 26.8 Review 历史被正式提升为放权学习的数据源
v1.3 最后进一步明确：
- review 历史不只是审计痕迹
- 它还是未来调整 Gate 阈值、扩大低风险自动 dispatch 范围的重要学习数据源
- 哪类动作常被批准、驳回、延后，本身就是联合认知体成长的重要治理资产

这意味着从 v1.3 开始，Gate 与 review 已经不再是两段分离设计，
而是被正式收束成“自动治理 + 人工治理 + 反馈学习”的同一条主线。

## 二十八、v1.4 的新增结论

### 28.1 v1.4 把 `SoulAction` 正式提升为“有完整生命旅程的治理对象”
v1.4 最关键的推进，是不再只把 `SoulAction` 当成认知层与执行层之间的中间动作对象，
而是正式把它理解为一个会经历提议、治理、执行、结果与回流全过程的生命周期对象。

也就是说：
- 它不是一次性函数调用
- 也不是生成即执行的轻命令
- 而是一条带有治理语义、执行语义与反馈语义的正式生命体

这意味着从这一版开始，联合认知体第一阶段第一次拥有了统一的动作生命周期观。

### 28.2 第一阶段的五段生命周期被正式压清
v1.4 进一步把 `SoulAction` 的完整旅程收束为五段：
1. 候选期（proposal phase）
2. 治理期（governance phase）
3. 执行期（execution phase）
4. 结果期（outcome phase）
5. 回流期（feedback reintegration phase）

这意味着第一阶段不再只是“对象生成动作，再执行动作”，
而是第一次明确动作必须经历从提出到回流的完整闭环旅程。

### 28.3 第一阶段的正式主状态集合与合法转移被确立
v1.4 进一步明确建议，第一阶段应正式承认以下主生命周期状态：
- `proposed`
- `gated`
- `pending_review`
- `approved`
- `deferred`
- `discarded`
- `dispatched`
- `running`
- `completed`
- `failed`
- `closed`

并且进一步压清了合法主转移链：
- `proposed` → `gated`
- `gated` → `dispatched` / `pending_review` / `deferred` / `discarded`
- `pending_review` → `approved` / `deferred` / `discarded`
- `approved` → `dispatched`
- `dispatched` → `running` / `completed` / `failed`
- `running` → `completed` / `failed`
- `completed` / `failed` / `discarded` → `closed`

这意味着后续实现中，Gate、Review、Dispatcher、执行层不应再各自发明状态语义，
而应被统一挂回同一条生命周期法典。

### 28.4 `completed` 与 `closed` 的区别被正式压实
v1.4 很关键地进一步明确：
- `completed` 只表示执行完成
- `closed` 才表示整个生命周期完成

也就是说：
- 一个动作执行结束后，还必须经过结果收束与反馈回流
- 只有当最小回流完成后，这条动作才真正从系统生命周期中闭合

这意味着联合认知体的闭环不再被理解成“执行结束”，
而是被理解成“执行结果重新成为认知输入之后的正式收束”。

### 28.5 主状态、审批状态、执行状态被正式要求分层理解
v1.4 进一步压实了一个很重要的工程判断：
- 主生命周期状态 ≠ 审批状态
- 审批状态 ≠ 执行状态

其中：
- 主生命周期状态负责描述动作整体旅程
- 审批状态负责描述治理资格
- 执行状态负责描述执行侧运行态

这意味着即使第一阶段暂时仍把这些信息压在同一张表里，
语义上也必须严格分层，否则系统很快会因为状态混用而失去可维护性。

### 28.6 生命周期所需的关键字段集合被正式压成六组
v1.4 还进一步把动作生命周期的关键物质载体收束为六组字段：
- 身份字段
- 候选依据字段
- Gate 字段
- Review 字段
- Dispatch / 执行字段
- 回流字段

其中最关键的推进是：
- `feedback_ref` 被正式提升为连接结果期与回流期的重要锚点
- `gate_decision`、`gate_reason_summary`、`risk_level` 等字段从此拥有了正式生命周期语义
- `reintegrated_at`、`continuity_effect`、`persona_effect`、`gate_learning_effect` 等字段被明确为后续阶段的正式目标

这意味着从 v1.4 开始，`soul_actions` 不再只是动作日志表，
而开始被理解为联合认知体生命周期记录层。

### 28.7 各组件在生命周期中的职责边界被正式钉住
v1.4 进一步明确了组件分工：
- 认知对象层负责提出候选
- `Intervention Gate` 负责判定资格
- Review 流负责人工治理
- Dispatcher 负责正式下发
- worker / Vault / OpenClaw / R2 / SQLite 负责实际执行与产物落地
- 回流层负责把结果重新解释为联合体内部认知变化

这意味着后续实现时，不能再把 Gate、Review、Dispatcher、执行侧的职责混写在一起，
而应围绕生命周期阶段做边界切分。

### 28.8 四条关键动作链的生命周期样板被正式收敛
v1.4 最后还把第一阶段最重要的四类动作压成了正式生命周期样板：
- 脑暴追问链
- Persona 更新链
- 连续性晋升链
- OpenClaw 桥接链

这意味着：
- 低风险交互动作、内部状态动作、高阈值连续性动作、外部执行桥接动作
- 终于都被挂回到同一条统一生命周期框架下理解

也为后续实现提供了最直接的参照模板。

### 28.9 回流期被正式确立为下一阶段的核心工程入口
v1.4 进一步明确：
- 回流不是附属补丁
- 它是联合认知体区别于普通自动化系统的核心差异层
- 一次动作的结果必须能回到 `PersonaState`、`EventNode`、`ContinuityRecord`、`InterventionDecision` 与后续动作生成中

这意味着 v1.4 实际上已经把下一步最自然的方向压出来了：
后续不应继续优先发明新动作名，

## 三十、v1.5 的新增结论

- v1.5 则正式回答了结果如何重新塑造联合体自身

这意味着从这一版开始，联合认知体第一阶段第一次拥有了“动作之后系统如何继续成长”的正式引擎层。

### 30.2 v1.5 正式确立了“结果不应直接等于记忆，而必须先经过再认知解释”的原则
v1.5 进一步明确：
- 执行结果不能直接等于长期记忆

这意味着回流引擎不只是“把结果写回去”，
而是正式承担解释、分流、定强度、留痕与触发下一轮动作的职责。

### 30.4 第一阶段的结果输入包被正式定义为“带治理语境的结果包”
v1.5 进一步明确：
- 回流引擎读取的不应只是 success / failure
- 而应是一包带有 `actionType`、`sourceType`、`gateDecision`、`approvalStatus`、`resultSummary`、`errorMessage`、`feedbackRef`、用户响应与产物引用的结果信号

也就是说：
- 回流层处理的不是孤立结果码
- 而是动作在完整治理与执行语境中的结果包

这意味着从 v1.5 开始，`feedback_ref`、`result_summary`、`error_message` 等字段第一次被正式提升为回流引擎的输入锚点。

### 30.5 五个正式回流目标被明确钉住
v1.5 进一步把反馈的主要去向正式收敛为五类：
1. `PersonaState`
2. `EventNode`
3. `ContinuityRecord`
4. `InterventionDecision`
5. 新一轮 `SoulAction` 候选生成

其中的关键判断是：
- Persona 更新应偏小步、累积式修正
- Event 回流应关注关键变化而不是普通结果
- Continuity 晋升应保持最高门槛
- InterventionDecision 回流主要服务于治理学习
- 结果还可能直接成为下一轮动作候选生成的起点

这意味着联合认知体第一次拥有了完整的“结果分流地图”。

### 30.6 回流强度分级被正式提出
v1.5 还进一步明确，反馈回流不应只是“写 / 不写”的二元判断，

- `light`
- `medium`
- `strong`
- `promotion`

这意味着回流引擎不仅要决定“去哪里”，
还必须决定“以多大力度去”。

也就是说：
- 轻量 Persona 微调
- 中度事件增强
- 强治理修正
- 长期连续性晋升

终于第一次拥有了统一的强度语义。

这意味着从这一版开始，回流不再只是统一口号，

- 但这些新动作不能绕开治理链
- 它们仍必须重新进入 Gate 与正式生命周期

这意味着系统虽然开始具备再生长能力，
但不会滑向失控的自激式递归动作链。

### 30.9 v1.5 把第一阶段真正的最小闭环压成了“输入 → 动作 → 执行 → 再认知 → 新动作 / 对象更新”
v1.5 最后进一步明确：

  - 输入
  - 对象
  - `SoulAction`
  - Gate
  - Review / Dispatch
  - 执行
  - 结果
  - 再认知
  - 对象更新 / 新动作生成

这意味着到 v1.5 为止，联合认知体第一阶段第一次真正拥有了可以持续塑造自己的 continuity integration 骨架。

v1.5 还直接给出了后续工程落点：

- 它应负责读取结果包、做再认知判断、写回对象层、并触发新动作候选

这意味着后续不应再优先发明更多概念名词，
而应继续进入回流引擎的代码落位、字段映射与最小实现顺序设计。

- 第一版继续不直接创建新动作

这意味着 `feedbackReintegration.ts` 第一版已经拥有了非常完整且克制的“解释器骨架”。
