> 起草日期：2026-03-24

---

## 一、角色定义

### 项目经理（PM）
- **载体**：专门的 AI 会话，负责宏观管控
- **职责**：
  - 维护 `vision/` 目录下全部文档
  - 审计进度、同步口径
  - 制定开发组任务优先级
  - 编写和下发开发任务单
  - 复盘和归档

### 开发组（DEV）
- **载体**：独立的 AI 会话，负责具体实现
- **职责**：
  - 接收并完成 PM 下发的任务
  - 遵守 `CLAUDE.md` 和 `tasks/lessons.md` 中的规则
  - 完成后更新任务状态
  - 本机编译验证后再提交

---

## 二、任务交接机制

### 2.1 任务下发流程

```
PM 复盘进度
  ↓
PM 编写任务单 → tasks/todo.md
  ↓
PM 在沟通纪要中记录决策
  ↓
用户确认任务优先级
  ↓
DEV 会话启动，读取 tasks/todo.md
  ↓
DEV 按任务执行
  ↓
DEV 完成后，在 tasks/todo.md 标记完成
  ↓
PM 下次复盘时验证
```

### 2.2 任务单格式（`tasks/todo.md`）

```markdown
# LifeOnline 开发任务

## 当前迭代：Sprint-YYYY-MM-DD

### P0 — 必须完成
- [ ] 任务描述
  - 目标：...
  - 关键文件：...
  - 完成标准：...
  - 验证方式：...

### P1 — 应当完成
- [ ] 任务描述
  - ...

### P2 — 可选
- [ ] 任务描述
  - ...
```

### 2.3 完成标记
- `[ ]` — 待做
- `[/]` — 进行中
- `[x]` — 已完成
- `[!]` — 阻塞（需 PM 或用户介入）

## 三、文件职责划分

| 文件/目录 | 谁写 | 谁读 | 用途 |
|---|---|---|---|
| `vision/00-权威基线/` | PM 审计 | DEV + PM | 方向性基线，不轻易修改 |
| `vision/01-当前进度/` | PM 更新 | DEV + PM | 进度对齐底板 |
| `vision/02-历史草案/` | PM 管理 | PM | 版本存档 |
| `vision/03-讨论纪要/` | PM 编写 | DEV + PM + 用户 | 沟通记录 |
| `tasks/todo.md` | PM 编写 → DEV 更新 | 双方 | 任务交接物 |
| `tasks/lessons.md` | DEV 更新 → PM 审核 | 双方 | 教训沉淀 |
| `CLAUDE.md` | PM/DEV | DEV | 开发环境上下文 |

---

## 四、进度同步节奏

### 4.1 日常节奏
- **DEV 每次会话结束时**：在 `tasks/todo.md` 标记完成项，简要写 result
- **PM 复盘时**：审计 `tasks/todo.md` 状态，同步更新 `vision/01-当前进度/`

### 4.2 阶段性节奏
- **每完成一个 P0/P1 任务**：PM 更新进度简表
- **每达成一个 PR 里程碑**：PM 更新正式任务书
- **每次重大决策**：PM 写入沟通纪要

---

## 五、开发组会话启动 Checklist

DEV 每次启动新会话时，应按以下顺序读取：

1. `CLAUDE.md` — 环境上下文
2. `tasks/lessons.md` — 历史教训
3. `tasks/todo.md` — 当前任务
4. `vision/01-当前进度/LifeOnline 第一阶段项目进度简表（快速查看版）.md` — 进度对齐

---

## 六、升级条件

以下情况需升级到用户决策：

1. 蓝图认知对象定义需要修改
2. PR 顺序需要调整
3. 新增蓝图外的 actionType
4. 架构边界变更（如：灵魂层从 LifeOS 独立出去）
5. 引入新技术栈
6. 删除已有功能

---

## 七、当前适用的约束

1. **编译先行**：本机 `npx tsc --noEmit` 通过后才能部署
2. **保守口径**：不把最小落地写成完整产品化系统
3. **review-backed**：高阈值动作必须经过治理链
4. **骨架优先**：增强联合认知体骨架的工作优于表层修补
