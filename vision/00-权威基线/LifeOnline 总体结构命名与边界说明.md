# LifeOnline 总体结构命名与边界说明

这份文档的目的，不是新增架构，而是把整个项目中几个最重要的名字和边界彻底对齐。

因为如果命名层级不清，后续所有关于“灵魂层”“联合体”“后台”“输入侧”“执行器”的讨论，都会不断发生语义漂移。

所以这份文档要回答的是：

> LifeOnline、LingGuangCatcher、LifeOS、Vault、OpenClaw、R2 这些名字分别代表什么？它们彼此是什么关系？边界在哪里？

---

# 1. 总体命名原则

## 1.1 `LifeOnline` 是整个项目的总名
`LifeOnline` 代表的是整个系统工程，含义是：

> **生命的在线延续。**

它不是某一个子模块，也不是某一个后端服务名，
而是整套联合体系统的总称。

在语义上，`LifeOnline` 指向的是：
- 用户真实生活流
- 输入采集系统
- 后台认知与执行系统
- 内容沉淀与展示系统
- 持续记忆与连续性基础设施
- 人与模型共同演化的长期工程

也就是说：

> 当我们在讨论整个愿景、整个联合体、整个“生命在线延续”工程时，应该优先使用 `LifeOnline`，而不是 `LifeOS`。

---

## 1.2 `LifeOS` 不是整个项目，而是其中一个核心子系统
`LifeOS` 是 `LifeOnline` 下面的一块核心系统，
它负责：
- 后台控制
- 索引与运行态
- worker task 编排
- schedule 触发
- 调用 OpenClaw
- 把分析与自动化结果写回
- 作为内容输出和控制台展板

所以 `LifeOS` 的正确定位应该是：

> `LifeOnline` 的后台与内容输出中枢。

它很重要，但它不是整个 `LifeOnline` 的全部。

---

# 2. LifeOnline 的两大主块

按照当前对齐，`LifeOnline` 下至少有两大主块：

```text
LifeOnline
├── LingGuangCatcher / 灵光 APP
└── LifeOS
```

---

## 2.1 `LingGuangCatcher` / 灵光 APP
### 定位
灵光 APP 是现实生活流的主输入侧。

### 主要职责
- 记录日常行为
- 承接即时灵感
- 接住头脑风暴
- 收集生活信号
- 形成原始输入流

### 在总体系统中的位置
它更接近：
- 传感层
- 采集层
- 输入侧入口

它的核心价值不是“做判断”，
而是把你的真实生活、真实行为、真实想法持续送入整个系统。

所以从联合体角度看：

> 灵光 APP 是 LifeOnline 的现实世界捕手。

---

## 2.2 `LifeOS`
### 定位
LifeOS 是后台系统，也是内容输出与控制展板。

### 主要职责
- 作为控制核心编排任务
- 维护 SQLite 运行态
- 管理 worker tasks / schedules
- 进行索引与查询
- 调用 AI 能力
- 调用 OpenClaw 作为外部执行器
- 输出日报、周报、总结、结构化 note 等内容
- 提供 Web 控制台

### 在总体系统中的位置
它更接近：
- 后台认知与编排系统
- 输出与展示系统
- 联合体的后台中枢

所以从联合体角度看：

> LifeOS 是 LifeOnline 的后台大脑与展板，而不是整个 LifeOnline 本身。

---

# 3. LifeOnline 里的关键基础设施边界

除了两大主块，还必须对几个经常被混淆的基础设施边界做明确说明。

---

## 3.1 `Vault`
### 定位
Vault 是内容事实源。

### 主要职责
- 保存 markdown 内容
- 保存 frontmatter 元数据
- 作为可读、可编辑、可回顾的内容沉淀层

### 不应被理解为
- 不应被理解为整个后台
- 不应被理解为主要调度器
- 不应被理解为完整运行态数据库

### 正确关系
在当前系统里：
- 内容事实 → Vault
- 运行控制 → SQLite / LifeOS backend

所以：

> Vault 是 LifeOnline 的内容记忆载体之一，但不是联合体的全部大脑。

---

## 3.2 `SQLite`
### 定位
SQLite 是索引与运行态存储。

### 主要职责
- notes 查询视图
- worker task 运行态
- schedule 运行态
- AI 配置
- 未来灵魂动作层运行态

### 不应被理解为
- 不应被理解为最终内容真相
- 不应被理解为人类主要阅读界面

所以：

> SQLite 是 LifeOS 的运行时骨架，不是 LifeOnline 全部意义上的记忆本体。

---

## 3.3 `OpenClaw`
### 定位
OpenClaw 是外部执行器。

### 主要职责
- 承接被委托的外部任务
- 处理适合远端或外部执行的复杂任务

### 不应被理解为
- 不应被理解为主编排器
- 不应被理解为系统主脑
- 不应被理解为任务真相持有者

### 正确关系
它的关系应该始终是：

```text
LifeOS 派发
   ↓
OpenClaw 执行
   ↓
结果回 LifeOS / Vault
```

所以：

> OpenClaw 是 LifeOnline 的手脚之一，不是中枢。

---

## 3.4 `R2`
### 定位
R2 是桥接、冷存储与备份基础设施。

### 主要职责
- 内网条件下的远程可达桥接
- 大体积材料中转
- 冷存储
- 连续性备份副本

### 不应被理解为
- 不应被理解为主运行态
- 不应被理解为主记忆体
- 不应被理解为主脑

所以：

> R2 是 LifeOnline 的远程基础设施之一，是桥，不是脑。

---

# 4. 正确的层级关系

当前更准确的层级关系应表述为：

```text
LifeOnline（生命的在线延续）
│
├── LingGuangCatcher / 灵光 APP
│     └── 现实生活流采集 / 日常行为记录 / 头脑风暴入口
│
├── LifeOS
│     ├── backend 控制核心
│     ├── SQLite 运行态 / 索引
│     ├── worker tasks / schedules
│     ├── AI 接入
│     ├── Web 控制台 / 内容输出展板
│     └── 灵魂层 / 认知内核（未来长在这里）
│
├── Vault
│     └── 内容事实源 / 可读沉淀层
│
├── OpenClaw
│     └── 外部执行器
│
└── R2
      └── 桥接 / 冷存储 / 连续性备份
```

这个图里最重要的点有两个：

## 4.1 `LifeOnline` 在最上层
因为它代表的是整个工程和总体愿景。

## 4.2 灵魂层当前更适合长在 `LifeOS` 内部
因为当前认知运行态、调度、worker、SQLite、输出展板都在 LifeOS 这一侧。

但它服务的目标，不是狭义的 `LifeOS` 后台本身，
而是整个 `LifeOnline` 联合体。

也就是说：

> 灵魂层当前的工程宿主是 LifeOS，但它面向的对象是整个 LifeOnline。

# 5. 后续文档里的推荐用词规则

为了避免之后继续混用，建议后续统一遵守下面的表述规则。

---

## 5.1 什么时候用 `LifeOnline`
当讨论这些内容时，用 `LifeOnline`：
- 整体愿景
- 联合体
- 人机共同演化系统
- 生命在线延续
- 输入侧 + 后台 + 输出侧 + 基础设施的整体关系

例如：
- “LifeOnline 的目标不是单点自动化，而是生命的在线延续。”
- “灵光和 LifeOS 都是 LifeOnline 的组成部分。”

---

## 5.2 什么时候用 `LifeOS`
当讨论这些内容时，用 `LifeOS`：
- backend
- worker tasks
- schedules
- SQLite
- Web 控制台
- 输出展板
- 当前灵魂层的工程宿主

例如：
- “LifeOS 是当前的后台控制核心。”
- “灵魂动作层第一阶段会先落在 LifeOS server 里。”

---

## 5.3 什么时候用 `LingGuangCatcher`
当讨论这些内容时，用 `LingGuangCatcher` / 灵光 APP：
- 日常行为记录
- 生活流输入
- 灵感采集
- 头脑风暴入口
- 真实世界信号捕获

例如：
- “LingGuangCatcher 是 LifeOnline 的现实生活输入端。”

---

# 6. 对当前灵魂层设计的直接影响

这个命名对齐不是表面工作，它会直接影响后面的设计表达。

---

## 6.1 灵魂层的正确表述应该更精确
更准确的说法不是：
- “给 LifeOS 注入灵魂”

而是：
- “在 LifeOS 这一后台宿主中实现 LifeOnline 联合体的认知内核”

这两个说法很像，但不一样。
前者容易把灵魂层误解成只是后台增强；
后者更清楚地说明：
- 工程落点在 LifeOS
- 服务对象是整个 LifeOnline

---

## 6.2 LingGuangCatcher 的战略地位会变高
如果没有这个命名对齐，容易把灵光 APP 看成只是输入工具。

但一旦从 `LifeOnline` 总体结构看，它其实是：
- 现实锚点
- 原始生活流采集器
- 灵魂层最重要的输入源之一

所以后续设计中，灵光不应被降格成“附属 App”，
而应被视为整个联合体的前端生命接口。

---

## 6.3 LifeOS 的表述会变得更干净
以后再谈 LifeOS，就可以更清楚地说：
- 它是后台
- 它是中枢
- 它是展板
- 它是当前灵魂层的宿主

但它不再背负“整个项目总名”的语义负担。

这会让后面所有 v1.0 之后的文档更清晰。

---

# 7. 一句话总结

> `LifeOnline` 是整个“生命在线延续”的联合体工程；`LingGuangCatcher` 是现实生活流的输入与采集侧；`LifeOS` 是后台控制核心与内容输出展板；Vault、SQLite、OpenClaw、R2 则分别承担内容真相、运行态、外部执行与桥接备份职责。后续所有灵魂层设计，都应在这个命名层级之上展开，而不再混用 `LifeOnline` 与 `LifeOS`。
